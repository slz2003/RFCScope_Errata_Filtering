## Issue 1 — ML-DSA-87 public key length mismatch (2592 vs 2602)

**Errata title:** Appendix A defines ML-DSA-87-PublicKey with the wrong SIZE

**Concise explanation:**  
Section 4 (and Table 2) define **ML-DSA-87 public keys as 2592 bytes**, but Appendix A’s ASN.1 module defines `ML-DSA-87-PublicKey ::= OCTET STRING (SIZE (2602))`. This conflicts with the main text, the parameter table, and even Appendix A’s own comment on `pk-ml-dsa-87` (“2592 octets”).

**Justification:**  (interop-breaking for ASN.1 users)  
- Any implementation/tooling that enforces the ASN.1 SIZE constraint from Appendix A will **reject valid 2592-byte ML-DSA-87 public keys** (or require meaningless padding).  
- This is a direct structural/quantitative contradiction inside the RFC and against the underlying parameter definition (FIPS 204 per experts).

**Minimal fix (erratum-style):**  
Change Appendix A ASN.1 line to:
- `ML-DSA-87-PublicKey ::= OCTET STRING (SIZE (2592))`
