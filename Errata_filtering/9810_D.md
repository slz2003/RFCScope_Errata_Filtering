## Issue 1 — Undefined `GenReqContent` and wrong PKIBody selector `genr` in D.5 request profile

**Errata title:** Appendix D.5 uses non-existent `genr (GenReqContent)` instead of `genm (GenMsgContent)` for PKI information request

**Concise explanation:**  
The D.5 *request* (genM) profile table says the PKIMessage body is `genr (GenReqContent)`, but the normative ASN.1 defines only `genm [21] GenMsgContent` (General Message) and `genp [22] GenRepContent` (General Response). There is no PKIBody alternative `genr` and no ASN.1 type `GenReqContent` in RFC 9810’s module. Section 6.5 also states the request MUST be `GenMsg` and the response MUST be `GenRep`, matching `genm/genp`.

**Justification:** inconsistency   
- A literal implementation of `genr (GenReqContent)` cannot compile against the RFC 9810 ASN.1.  
- The intended encoding is unambiguous from the ASN.1 and the surrounding text: request uses `genm (GenMsgContent)`, response uses `genp (GenRepContent)`.

**Minimal patch suggestion:**  
In Appendix D.5, under the genM (request) profile, change:

- **OLD:** `body  genr (GenReqContent)`  
- **NEW:** `body  genm (GenMsgContent)`

(Keeping the “GenMsgContent empty SEQUENCE” line as-is is fine; optionally clarify it means an empty `SEQUENCE OF InfoTypeAndValue`.)

**Impacted dimensions:** Structural, Terminology, CrossRFC, Causal

**Experts that flagged it:** Causal, Structural, Terminology, CrossRFC
